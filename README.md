## Raylua

[LuaJIT](https://luajit.org/)-based bindings for [raylib](https://www.raylib.com/), a simple and easy-to-use library for learning video game programming.

This project is based on [TSnake41/raylib-lua](https://github.com/TSnake41/raylib-lua). And attempts to provide, development wise, a similar experience to
LÖVE2D... but for Raylib... with 3D support :D 

## Usage

raylua provides two binaries, inspired by LÖVE:

| Binary    | Purpose      | Windows Console   | Linux/macOS Console |
| --------- | ------------ | ----------------- | ------------------- |
| `raylua`  | Distribution | Hidden by default | Visible             |
| `rayluac` | Development  | Always visible    | Visible             |

### `raylua`

The main binary for running packaged or distributed games.

```sh
# Run a .rlua game package (zip file)
./raylua game.rlua

# Run a single Lua file
./raylua main.lua

# Run a game folder
./raylua mygame/
```

### `rayluac`

The developer-oriented binary for testing and debugging.

```sh
# Run the Lua REPL (interactive shell)
./rayluac --shell

# Run a single Lua file with console output
./rayluac main.lua

# Run a game folder with console output
./rayluac mygame/
```

On Windows, `raylua` hides the console window by default, while `rayluac` keeps it visible. On Linux and macOS, both run in the terminal.

## Building

Initialize submodules first:

```sh
git submodule update --init --recursive
```

Then build everything:

```sh
make
```

This builds:

- LuaJIT
- raylib
- generated binding code
- `raylua`
- `rayluac`

### Useful build targets

```sh
make luajit        # build LuaJIT only
make raylib        # build raylib only
make bindings      # regenerate tools/*.h binding snapshots
make autocomplete  # regenerate autocomplete API files
make update        # regenerate bindings + autocomplete
make clean         # remove build outputs
```

### Notes

A working Lua interpreter is needed for the code generation steps. By default, the LuaJIT interpreter built in this repository is used.

You can override it with:

```sh
LUA=luajit/src/luajit make
```

When cross-compiling, you may want to point `LUA` to a host interpreter that runs on your build machine.

## Updating raylib and regenerating bindings

When updating to a newer raylib version, the usual workflow is:

### 1. Update submodules

```sh
git submodule sync --recursive
git submodule update --init --remote --merge raylib raygui physac
```

### 2. Regenerate binding snapshots

```sh
make bindings
```

This regenerates the binding snapshot files in `tools/`.

You can also run the generator directly:

```sh
luajit/src/luajit tools/update_bindings.lua
```

Or update a single module:

```sh
luajit/src/luajit tools/update_bindings.lua api
luajit/src/luajit tools/update_bindings.lua raymath
luajit/src/luajit tools/update_bindings.lua rlgl
luajit/src/luajit tools/update_bindings.lua gestures
luajit/src/luajit tools/update_bindings.lua rcamera
luajit/src/luajit tools/update_bindings.lua physac
luajit/src/luajit tools/update_bindings.lua raygui
```

### 3. Regenerate autocomplete files

Autocomplete metadata is generated separately using raylib's parser:

```sh
make autocomplete
```

This writes API metadata into:

```txt
autocomplete/api/
```

If the parser is not built yet, build it with:

```sh
make -C raylib/tools/rlparser rlparser
```

### 4. Rebuild

```sh
make clean
make
```

### 5. Update `src/raylib.lua` if needed

The generated bindings cover function declarations, but `src/raylib.lua` still contains manually maintained FFI type definitions and helpers.

If upstream adds or changes:

- structs
- enums
- callback typedefs
- convenience helpers

you may still need to update `src/raylib.lua`.

## Loading embedded resources

raylib-lua supports loading resources from payloads using the normal raylib API.

You can also load arbitrary files from payloads using `raylua.loadfile`, which returns the file contents or an error.

## Creating structs

raylib structs are created using LuaJIT FFI:

```lua
local ffi = require "ffi"
```

Then create values with `ffi.new`, for example:

```lua
local color = ffi.new("Color", 255, 0, 0, 255)
```

Many raylib functions do not require manual struct creation, so you only need this when working with APIs that expect explicit C data.

### Note on pointers

You can use `rl.ref` to build a pointer from a cdata struct.

`rl.ref` only works with struct cdata. For primitive values, create a one-element array instead:

```lua
local int_ptr = ffi.new("int[1]")
local data = tostring(rl.LoadFileData("test.txt", int_ptr))
local count = tonumber(int_ptr[0])
```

## Simple test example

```lua
rl.SetConfigFlags(rl.FLAG_VSYNC_HINT)

rl.InitWindow(800, 450, "raylib [core] example - basic window")

while not rl.WindowShouldClose() do
  rl.BeginDrawing()

  rl.ClearBackground(rl.RAYWHITE)
  rl.DrawText("Congrats! You created your first window!", 190, 200, 20, rl.LIGHTGRAY)

  rl.EndDrawing()
end

rl.CloseWindow()
```

## Global API

If you want the raylib binding available through globals instead of the `rl` table, add this near the top of your script:

```lua
setmetatable(_G, { __index = rl })
```

There is an example of this in `lua_global_api.lua`.

## Editor support

There is limited autocomplete support for VS Code and other EmmyLua / LuaLS-compatible frontends using the generated files in `autocomplete/api/`.

For Lua Language Server, see the [LuaLS custom libraries documentation](https://github.com/LuaLS/lua-language-server/wiki/Libraries#custom).

It may also be possible to get autocomplete working in [ZeroBrane Studio](https://github.com/pkulchenko/ZeroBraneStudio).

## Debugging

You can use [Local Lua Debugger for Visual Studio Code](https://marketplace.visualstudio.com/items?itemName=tomblind.local-lua-debugger-vscode) for debugging support in VS Code.

Add this near the beginning of your script:

```lua
do local f = getmetatable(rl).__index; rawset(rl, "__index", function(_, k) return select(2, pcall(f, _, k)) end) end
package.path = package.path .. os.getenv("LUA_PATH")
local lldebugger = require "lldebugger"
lldebugger.start()
```

You also need a launch configuration that runs `rayluac`, for example:

```json
{
  "type": "lua-local",
  "request": "launch",
  "name": "(Lua) Launch",
  "cwd": "${workspaceFolder}",
  "program": { "command": "PATH TO rayluac" },
  "args": ["main.lua OR ${file} OR WHATEVER"]
}
```

This debugger does not support pausing at arbitrary unspecified points. You need to place a breakpoint before execution, or trigger an error to break into the debugger.

It also adds noticeable overhead, so expect reduced performance in heavier projects.
